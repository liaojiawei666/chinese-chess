# 文档撰写风格

本规范约定本仓库技术文档（`.md` / `.mdc`）的撰写风格，目标是**简洁、图文公式结合、易看懂**。

配图的**生成与导出流程**见 [`document-diagram.md`](./document-diagram.md)；本文只约定**风格**，配图的**视觉风格**见第 5 节。

## 1. 总体原则

- **图 + 公式承载主要信息，文字只做精简串联**。能用图、表、公式说清的，就不写大段文字。
- 面向"看懂概念"，不堆术语；一句话讲一个点。
- 先给全局，再展开细节（总—分）。

## 2. 结构与组织

- 先放总览（一张"地图"图或一段概述），后续每节是它的局部放大。
- 概念性内容按**自底向上、先前向后反向**组织（如网络：先输入/各层，再损失/反向）。
- 每节结构尽量统一：**一张图 + 一条核心公式 + 2~4 行文字**。
- 章节标题用名词短语或问题句，顺序与代码 / 规则的执行顺序一致。

## 3. 语言与措辞

- 中文叙述；保持简短，多用要点列表，少用长段落。
- 代码标识符（变量、函数、类名）一律用**英文，不用中文拼音**；正文中用反引号标注，如 `PolicyValueNet`。
- 不写"代码锚点"式逐行绑定（容易随代码漂移）；确需关联代码时，只点到函数 / 类名。

## 4. 公式

- 关键公式用块级 `$$ … $$`，行内量用 `$…$`。
- 公式后紧跟符号说明，**只解释非显然的量**，不逐符号复述。

## 5. 配图风格

> 生成步骤、XML 良构、导出命令见 [`document-diagram.md`](./document-diagram.md)；本节只约定"长什么样"。

### 5.1 形状语义（靠形状区分"是什么"）

- **张量 / 数据** → 立方体 `shape=cube`
- **运算层** → 圆角矩形 `rounded=1`
- **容器 / 分组**（如"主干""残差块内部"）→ 虚线大框 `dashed=1`，标题居左上 `verticalAlign=top`
- **运算符**（如残差相加 ⊕）→ 椭圆 `ellipse`
- **关键说明** → 底部整条说明框

### 5.2 配色（语义化，固定一套）

| 用途 | 填充 / 描边 |
| --- | --- |
| 红方 / value 头 | `#f8cecc` / `#b85450` |
| 黑方 / policy 头 | `#dae8fc` / `#6c8ebf` |
| 主干 / 特征张量 | `#d5e8d4` / `#82b366` |
| 输入张量 / 提示 | `#fff2cc` / `#d6b656` |
| 强调符号（跳连、⊕） | `#ffe6cc` / `#d79b00` |
| 关键说明框 | `#dae8fc` / `#6c8ebf` |

### 5.3 标题、标签与说明

- 画布 `adaptiveColors="auto"`，`background=#ffffff`。
- 标题置顶、加粗、`fontSize=16~17`，格式"图：……（一句副标题点明记号含义）"。
- 节点文字**中文、短**。
- 箭头标签用**白底** `labelBackgroundColor=#ffffff`，颜色跟随所属分支。
- 图底部放一条**关键说明框**，用一两句话总结全图要点。

## 6. 文件与元数据约定

- 知识 / 规范类 `.mdc` 设 `alwaysApply: false`，按需引用，不常驻上下文。
- 图源文件放 `docs/diagrams/`，导出的 `.svg` 放 `docs/diagrams/exports/`，二者分离。
- 文件名用 kebab-case，名字能反映内容。
- Markdown 中用相对路径引用导出图，如 `![标题](./diagrams/exports/<name>.svg)`。
